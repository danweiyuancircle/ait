# AI 工程化实践：从规范约束到复合增长

> 来源：[知乎 - 认知重建：Speckit用户的AI工程化实践总结](https://zhuanlan.zhihu.com/p/1993009461451831150)

## 核心观点

**AI 的决策质量 ∝ 可用信息的完整性 × 信息的信噪比**

不是越多信息越好，而是"高信号、低噪声"的精准信息在对的时刻主动推送。本质是把知识编码进工具，让工具在正确的时刻主动提醒，而非写文档让人看。

## 三个认知转变

### 1. 从"规范约束 AI"到"完整信息助力 AI"

- Speckit 思路：规范化流程 → 约束 AI 行为
- AI 工程化思路：上下文完整性 → AI 决策质量 → 自动沉淀经验 → 下次更好

### 2. 从"编写 Spec"到"沉淀 Context"

Context 分层管理，按阶段按需加载，而非一次性加载所有信息：

```
context/
├── business/       # 业务背景
├── tech/           # 技术背景
└── experience/     # 历史经验
```

### 3. 从"单次执行"到"复合增长"

| 执行次数 | 传统方式 | AI 工程化 | 累计节省 |
|---------|---------|----------|---------|
| 第 1 次 | 45 分钟 | 45 分钟 | 0 |
| 第 2 次 | 45 分钟 | 15 分钟 | 30 分钟 |
| 第 5 次 | 45 分钟 | 5 分钟 | 130 分钟 |
| 第 10 次 | 45 分钟 | 3 分钟 | 315 分钟 |

知识留在 context/ + skill/，支持边际成本递减。

## 技术架构

### 整体流转

```
用户输入 → Command(入口) → Agent(决策层) → Skill(执行层)
                ↓
          意图识别、流程路由、上下文管理
                ↓
          调用具体 Skill 执行
```

### 目录结构（位置即语义）

```
your-project/
├── AGENTS.md           # 项目记忆入口
├── .codebuddy/
│   ├── agents/         # Agent 定义
│   ├── commands/       # 命令入口（极简，仅 2 个）
│   └── skills/         # Skill 定义
├── context/            # 项目知识库（分层）
│   ├── business/       # 业务背景
│   ├── tech/           # 技术背景
│   └── experience/     # 历史经验
└── requirements/       # 需求管理
```

**设计原则**：极简主义入口，整个系统只有 2 个命令（`/req-dev`、`/optimize-flow`），复杂性下沉到 Agent 的智能路由。

## 经验沉淀的三层演进

```
阶段 1：纯文档（被动）
  └─ context/experience/商品发放-钱包选择问题.md

阶段 2：校验 Skill（半自动）
  └─ skill/product-distribution-validator 自动检查

阶段 3：完整 Command（全自动）
  └─ cmd/implement-product-distribution 一键完成
```

每一次踩坑都从"人要记住"逐步升级为"系统自动处理"。

## Subagent 架构解决上下文爆满

借鉴 Anthropic 的多 Agent 架构：

```
传统模式：单一 Agent → 上下文持续膨胀 → 窗口爆满

Subagent 模式：
  主 Agent(决策) → [Subagent1 | Subagent2 | Subagent3]（各独立窗口）
                          ↓ 结构化状态文件传递 ↓
                        主 Agent 继续（结果汇总）
```

效果对比：

| 指标 | 传统模式 | Subagent 模式 |
|------|---------|--------------|
| 窗口爆满频率 | 70% | 5% |
| 任务完成率 | 60% | 95% |
| 上下文利用效率 | 30% | 80% |

关键：Subagent 通过结构化文件交接状态，不强依赖"记忆"。

## 上下文工程三原则

1. **分层式信息组织** — 按阶段按需加载，非一次性全量加载
2. **即时上下文策略** — 维护轻量索引 → 运行时动态加载 → 窗口始终精简
3. **工具返回高信号信息** — 不依赖"记忆"，依赖结构化状态

## 复合工程核心理念

> "每一单元的工程工作应使后续工作变得更容易，而非更难"

Plan → Work → Review → Compound 的闭环：每个功能都产出文档模式、创建可复用组件、建立减少决策疲劳的约定。
